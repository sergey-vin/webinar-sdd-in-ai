# Spec-Driven Development (SDD)

Легковесный фреймворк для синхронизации продуктовых спецификаций, технической документации и кода — особенно полезен при работе с AI-агентами.

## Зачем

Без спецификаций AI-агенты выдумывают фичи, неправильно понимают скоуп и пишут код, который не вписывается в проект. Без верификации спецификаций документация расходится с реальностью. SDD решает обе проблемы: спецификации направляют разработку, код верифицирует спецификации.

## Три слоя

```
docs/                          # Конституция + Документация реализации
├── mission.md                 #   КТО пользователи, ЗАЧЕМ это существует
├── tech-stack.md              #   КАКИЕ технологии, КАК они связаны
├── roadmap.md                 #   КОГДА — завершённые фазы + планы
├── auth.md                    #   Детальная техническая документация
├── payments.md                #   (по одному на технический домен)
└── ...

specs/                         # Спецификации путей пользователя (планирование)
├── 2026-01-onboarding/
│   ├── plan.md                #   Чеклист: что сделано [x], что нет [ ]
│   ├── requirements.md        #   User story, скоуп, решения, известные пробелы
│   └── validation.md          #   Критерии приёмки + QA-сценарии
├── 2026-02-booking/
│   └── ...
└── ...
```

### Слой 1: Конституция (`docs/mission.md`, `tech-stack.md`, `roadmap.md`)

Идентичность проекта. Меняется редко. Отвечает на: что это, для кого, какой стек, какой план и что мы намеренно не делаем (не-цели).

### Слой 2: Спецификации путей (`specs/YYYY-MM-название/`)

По одной директории на **путь пользователя** — не на технический модуль. «Экран чата» — это не фича; это инструмент, используемый в разных путях (переговоры, поддержка, разрешение споров).

Каждый путь содержит 3 файла:

- **plan.md** — Чеклист задач, сгруппированных по высокоуровневым целям. `[x]` сделано, `[ ]` не сделано.
- **requirements.md** — User story, скоуп, не-цели, архитектурные решения, известные пробелы. Ссылки на документацию реализации.
- **validation.md** — Критерии приёмки + детальные QA-сценарии с чекбоксами.

### Слой 3: Документация реализации (`docs/*.md`)

Детальная техническая документация по доменам (авторизация, платежи, чат и т.д.). Это справочный материал — спецификации ссылаются на него, а не дублируют.

**Ключевое правило:** Документация описывает то, что **существует**. Спецификации отслеживают то, что **запланировано и не реализовано**.

## Рабочий процесс

### Новый проект (greenfield)

1. **Конституция** — Запустить `prompts/01-constitution.md` для создания mission, tech-stack, roadmap
2. **Спецификации путей** — Запустить `prompts/02-journey-specs.md` для каждого пути из roadmap
3. **Разработка** — Использовать спецификации для направления разработки
4. **Верификация** — Запустить `prompts/04-verify-specs.md` после реализации для поиска расхождений

### Существующий проект (brownfield)

1. **Аудит** — Запустить `prompts/03-retrofit-audit.md` для извлечения SDD из существующего кода
2. **Ревью** — Исправить неточности, которые аудит пропустил
3. **Продолжение** — Использовать `prompts/02-journey-specs.md` для новых фич

## Пути, а не модули

Это самый важный принцип. Организуйте спецификации вокруг того, что делает **пользователь**, а не как устроен **код**.

Плохо (по модулям):

```
specs/auth/          # Технический модуль
specs/chat/          # Технический модуль
specs/payments/      # Технический модуль
```

Хорошо (по путям пользователя):

```
specs/signup-and-verify/          # Регистрация, верификация, доверие
specs/create-and-search/          # Публикация, просмотр, поиск совпадений
specs/negotiate-and-pay/          # Чат, переговоры, бронирование, оплата
specs/fulfill-and-track/          # Передача, отслеживание, подтверждение
specs/review-and-support/         # Оценка, жалоба, поддержка, возврат
```

Чат появляется в нескольких путях (переговоры в negotiate-and-pay, поддержка в review-and-support). Платежи охватывают negotiate-and-pay (первичная оплата) и review-and-support (возвраты). Организация по путям естественно отражает это.
