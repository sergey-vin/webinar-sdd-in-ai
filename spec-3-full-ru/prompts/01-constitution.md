# Промпт: Создание конституции SDD

## Инструкции для AI-агента

Ты создаёшь конституцию SDD (Spec-Driven Development) для проекта. Она состоит из 3 файлов в директории `docs/` (или `specs/`, если проект предпочитает):

1. **`mission.md`** — Что это за проект, для кого и зачем он существует
2. **`tech-stack.md`** — Какие технологии используются и как они связаны
3. **`roadmap.md`** — Что уже сделано и что запланировано

## Прежде чем писать

Используй `AskUserQuestion` для уточнения:

### Для mission.md

- Что делает продукт в одном предложении?
- Кто основные пользователи/персоны? (2-4 роли)
- Какую проблему решает? Почему нельзя использовать существующие решения?
- Какая бизнес-модель? (Кто платит, за что?)
- Есть ли ключевые принципы или ограничения? (например, «mobile-first», «работает офлайн», «платит только продавец,долю с каждой оплаты, а для покупателя это бесплатно»)
- Что продукт намеренно НЕ делает? Что исключено из MVP? (например, «нет веб-версии», «нет реал-тайм отслеживания», «нет мультиязычности»)

### Для tech-stack.md

- Какой фронтенд? (Фреймворк, управление состоянием, ключевые библиотеки)
- Какой бэкенд? (Фреймворк, база данных, ORM, авторизация)
- Какая инфраструктура? (Хостинг, CI/CD, мониторинг)
- Есть ли важные архитектурные решения? (Почему X, а не Y?)

### Для roadmap.md

- Какие фичи уже выпущены? (Сгруппировать по логическим фазам)
- Что запланировано дальше? (Будущие фазы в порядке приоритета)
- Есть ли известные пробелы в существующих фичах?
- Какие фичи явно исключены? (Не «ещё не сделали», а «не будем делать» или «не для MVP»)

## Структура файлов

Используй шаблоны из `sdd/templates/` как отправную точку. Ключевые правила:

- **mission.md** начинается с «Что/Зачем» простым языком. Без жаргона. Нетехнический стейкхолдер должен понять.
- **tech-stack.md** перечисляет технологии с однострочными описаниями. Ссылки на детальную документацию где она есть. Ключевые архитектурные решения в конце.
- **roadmap.md** использует `[x]` для завершённых пунктов, `[ ]` для запланированных. Каждая фаза имеет короткое название и ссылки на документацию. Названия фаз — короткие.

## Пример

Смотри `sdd/templates/mission.md`, `sdd/templates/tech-stack.md`, `sdd/templates/roadmap.md` для структуры.

## После создания файлов

- Перечитай все 3 файла и проверь внутреннюю согласованность
- Убедись, что персоны из mission.md соответствуют фичам из roadmap
- Убедись, что tech-stack.md покрывает всё, упомянутое в roadmap
- Попроси пользователя проверить перед переходом к спецификациям путей
