# Промпт: Создание спецификаций пути пользователя

## Инструкции для AI-агента

Ты создаёшь спецификации для пользовательского пути. Каждый путь получает директорию в `specs/` с 3 файлами:

```
specs/YYYY-MM-название/
├── plan.md            # Что сделано, что нет
├── requirements.md    # Скоуп, решения, пробелы
└── validation.md      # Как проверить, что всё работает
```

## Прежде чем писать

1. Прочитай `docs/mission.md` и `docs/roadmap.md` для понимания контекста
2. Определи следующий неспецифицированный путь из roadmap
3. Используй `AskUserQuestion` для уточнения:
   - Что пользователь пытается сделать? («Работа, которую нужно выполнить»)
   - Какие ключевые шаги в этом пути?
   - Какие решения уже приняты? Что было отвергнуто и почему?
   - Есть ли граничные случаи или известные проблемы?
   - Что явно исключено из этого пути? (Не-цели для MVP)
   - Что должна проверить QA-команда для подтверждения работоспособности?

## Критическое правило: Пути, а не модули

Путь — это то, что делает **пользователь**, а не как устроен **код**.

- ПЛОХО: «Фича чата», «Модуль платежей», «Система авторизации»
- ХОРОШО: «Договориться и забронировать» (пользователь ведёт переговоры в чате, потом платит для фиксации сделки)

Один путь часто затрагивает несколько технических модулей. Это нормально — спецификация описывает опыт пользователя и ссылается на техническую документацию для деталей реализации.

## Структура файлов

### plan.md

```markdown
# Название пути — План

Цель пользователя: **Одно предложение, описывающее что пользователь хочет достичь.**

## Высокоуровневая цель 1

- [x] Завершённая задача
- [x] Ещё одна завершённая задача
- [ ] Запланировано, но не реализовано

## Высокоуровневая цель 2

- [x] ...
```

Правила:

- `[x]` = реализовано и проверено в коде
- `[ ]` = запланировано, но ещё не сделано
- Группировка по пользовательским целям, не по техническим слоям
- Первая строка после заголовка: цель пользователя жирным
- Пункты достаточно конкретны для проверки, но достаточно коротки для беглого просмотра

### requirements.md

```markdown
# Название пути — Требования

## User Story

> Как [персона], я хочу [действие], чтобы [результат].

## Скоуп

Один абзац, определяющий границу этого пути.

## Не-цели

- Что этот путь намеренно НЕ покрывает
- Фичи, которые пользователи могут ожидать, но исключены из MVP

## Детали реализации

- Фича A: [docs/feature-a.md](../docs/feature-a.md)
- Фича B: [docs/feature-b.md](../docs/feature-b.md)

## Решения

- **Название решения** — что решили и почему
- **Другое решение** — контекст выбора

## Известные пробелы

- Описание пробела (что отсутствует или сломано)

## Отвергнутые альтернативы

- **Альтернатива** — почему отвергнута
```

Правила:

- User stories используют имена персон из `mission.md`
- Детали реализации ссылаются на существующую документацию — не дублировать
- Решения фиксируют «почему», а не только «что»
- Известные пробелы — честно, не скрывать проблемы
- Нереализованные пункты получают явные маркеры

### validation.md

```markdown
# Название пути — Валидация

## Критерии приёмки

- [ ] Высокоуровневая проверка 1
- [ ] Высокоуровневая проверка 2

---

## Фича — QA-сценарии

### Предусловия

Что нужно подготовить перед тестированием.

### 1. Название сценария

1. Шаг 1
2. Шаг 2

- [ ] Ожидаемый результат
- [ ] Ещё одна проверка
```

Правила:

- Верхняя секция: критерии приёмки (что проверяет PM)
- Нижняя секция: детальные QA-сценарии (что выполняет тестировщик)
- Включать предусловия (тестовые аккаунты, подготовка данных)
- Включать граничные случаи и сценарии ошибок
- Критические баги отмечать явно

## После создания файлов

- Сверить пункты plan.md с решениями из requirements.md
- Убедиться, что validation.md покрывает каждый пункт из plan.md
- Обновить `docs/roadmap.md` если путь выявил новые фазы или пробелы
- Обновить `docs/mission.md` или `docs/tech-stack.md` если появился новый контекст
