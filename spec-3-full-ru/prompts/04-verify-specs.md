# Промпт: Верификация спецификаций по коду

## Инструкции для AI-агента

Ты проверяешь, что SDD-спецификации соответствуют реальной кодовой базе. Спецификации расходятся с реальностью — фичи отмечаются как сделанные, хотя это не так, значения enum меняются, логика таймингов отличается от задокументированной.

## Процесс

### 1. Запуск параллельных агентов исследования

Для каждого пути в `specs/` запусти агента, который:

- Читает `plan.md`, `requirements.md`, `validation.md` этого пути
- Находит соответствующий код (маршруты бэкенда, экраны фронтенда, модели базы)
- Проверяет **каждое утверждение** в спецификациях по реальному коду

### 2. Что проверять

Для каждого `[x]` пункта в `plan.md`:

- Фича действительно существует в коде?
- Работает так, как описано?

Для каждого решения в `requirements.md`:

- Отражено ли оно в коде? (например, «списание при оформлении» — проверить обработчик маршрута)
- Правильны ли значения enum? (например, типы категорий, значения статусов)

Для каждого утверждения в `validation.md`:

- Проверяемо ли ожидаемое поведение из кода?

Для документации реализации (`docs/*.md`):

- API-эндпоинты совпадают с определениями маршрутов?
- Модели данных совпадают с файлами миграций?
- Описания потоков совпадают с реальным потоком кода?

### 3. Типичные неточности

- **Фичи отмечены как реализованные, но их нет** (соцсети для входа, 2FA и т.д.)
- **Неправильные значения enum** (например, `premium` в документации, но `pro` в коде)
- **Неправильные тайминги** (например, «24ч после завершения», но код использует «следующий крон после смены статуса»)
- **Неправильный поток** (например, «списание при подтверждении», но код списывает при оформлении)
- **Отсутствующий контекст безопасности** (например, «2-шаговая смена email», но реально только 1 шаг)
- **Преувеличенные возможности админки** (например, «админ может переоткрыть тикет», но есть только закрытие)

### 4. Исправление неточностей

Для каждой найденной неточности:

1. Исправить файл спецификации в соответствии с реальностью кода
2. Если это реальный пробел (фича должна быть, но её нет), отметить как `[ ]` в plan.md и добавить в «Известные пробелы» в requirements.md
3. Исправить документацию реализации, если она содержит ту же ошибку
4. Обновить docs/roadmap.md, если пробел влияет на завершённость фазы

### 5. Формат отчёта

Для каждого пути:

```
## Название пути — Отчёт верификации

### Подтверждено (верно)
- Утверждение 1 — подтверждено в файл:строка
- Утверждение 2 — подтверждено в файл:строка

### Найденные и исправленные неточности
- Утверждение → Что было в спецификации → Что делает код → Применённое исправление

### Не поддаётся проверке
- Утверждения, которые не удалось проверить (например, поведение, зависящее от времени)
```

## Советы

- **Читай код, а не комментарии.** Комментарии врут. Код — нет.
- **Проверяй и бэкенд, и фронтенд.** Наличие эндпоинта на бэкенде не значит, что UI его вызывает.
- **Проверяй и админку тоже.** Возможности админки часто преувеличены в спецификациях.
- **Запускай агентов параллельно.** По одному на путь. Объединяй результаты в конце.
- **Будь конкретен.** «Неправильное значение enum» — бесполезно. «Тип категории `premium` в документации, но `pro` в коде (файл миграции X, строка Y)» — можно сразу исправить.
