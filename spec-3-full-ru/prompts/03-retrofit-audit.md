# Промпт: Внедрение SDD в существующий проект

## Инструкции для AI-агента

Ты извлекаешь фреймворк SDD (Spec-Driven Development) из существующей кодовой базы, в которой нет спецификаций. Цель — задокументировать **то, что реально существует** — не то, что было запланировано.

## Фаза 1: Обнаружение

Просканируй кодовую базу для составления инвентаря фич:

### Бэкенд

- Список всех API-маршрутов (сгруппировать по ресурсу/домену)
- Список всех моделей/таблиц базы данных
- Список всех фоновых задач, кронов, вебхуков
- Список всех внешних интеграций (платежи, авторизация, email и т.д.)

### Фронтенд

- Список всех экранов/страниц (из конфигурации роутера)
- Список всех стейт-менеджеров (stores/blocs/cubits)
- Список всех API-вызовов (из слоя сервисов/репозиториев)

### Админка (если есть)

- Список всех страниц админки
- Доступные CRUD-операции

## Фаза 2: Маппинг на пути пользователя

Сгруппируй найденные фичи в **пользовательские пути**, а не в технические модули.

Спроси пользователя: «На основе найденного, я вижу такие пользовательские пути: [список]. Совпадает ли это с вашим представлением о продукте?»

Типичные паттерны путей:

- Онбординг (регистрация, верификация, настройка профиля)
- Основное действие (создание контента, поиск совпадений)
- Транзакция (переговоры, договорённость, оплата)
- Исполнение (доставка, подтверждение, отслеживание)
- Пост-транзакция (отзыв, спор, поддержка)

## Фаза 3: Создание конституции

### mission.md

- Извлечь персоны из кода (роли пользователей, роли админа)
- Определить бизнес-модель из логики платежей
- Описать что делает продукт на основе найденных фич
- Спросить пользователя про «Зачем» — код не объясняет мотивацию

### tech-stack.md

- Извлечь из package.json, pubspec.yaml, docker-compose и т.д.
- Перечислить фреймворки, базы данных, внешние сервисы с версиями
- Задокументировать ключевые архитектурные решения, видимые в коде

### roadmap.md

- Всё найденное в коде идёт в «Завершено» с `[x]`
- Спросить пользователя что запланировано для секций «Предстоящее»
- Сгруппировать завершённые пункты по логическим фазам

## Фаза 4: Создание спецификаций путей

Для каждого пути создать `plan.md`, `requirements.md`, `validation.md`:

- **plan.md**: Отметить всё найденное в коде как `[x]`. Отмечать `[ ]` только для пунктов, которые пользователь явно назвал запланированными.
- **requirements.md**: Задокументировать решения, видимые в коде (например, «оплата списывается при оформлении, а не при подтверждении» — определено из потока кода). Спросить пользователя о решениях, не видимых в коде.
- **validation.md**: Написать критерии приёмки на основе того, что код делает. Добавить QA-сценарии для самых критичных потоков.

## Фаза 5: Создание документации реализации

Для каждого технического домена достаточной сложности создать `docs/{домен}.md`:

- Начать с **обзора простым языком**: что делает, для кого, как текут деньги/данные
- Затем добавить технические детали: API-эндпоинты, модели данных, диаграммы потоков
- Документация описывает то, что **существует**. Нереализованное идёт в спецификации.

## Критические правила

1. **Документируй реальность, а не мечты.** Если код этого не делает — не отмечай `[x]`.
2. **Проверяй утверждения по коду.** Не доверяй комментариям, именам переменных или старой документации. Читай реальную логику.
3. **Отмечай противоречия.** Если старая документация говорит одно, а код делает другое — зафикси в «Известных пробелах».
4. **Спрашивай пользователя.** Код объясняет «что» и «как», но редко «зачем» и «что дальше». Часто используй `AskUserQuestion`.
5. **Запускай параллельные агенты исследования** для больших кодовых баз. По одному на путь или домен. Объединяй результаты.

## Структура вывода

```
docs/
├── mission.md
├── tech-stack.md
├── roadmap.md
├── {домен-1}.md
├── {домен-2}.md
└── README.md          # Индекс всей документации

specs/
├── {путь-1}/
│   ├── plan.md
│   ├── requirements.md
│   └── validation.md
├── {путь-2}/
│   └── ...
└── ...
```
